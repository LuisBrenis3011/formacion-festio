================================================================================
DOCUMENTACIÓN DEL FLUJO INTELIGENTE DE RECOMENDACIONES - FESTIO
================================================================================

Este documento detalla el comportamiento del "Cerebro" de Festio: el motor 
de búsqueda y recomendación inteligente de paquetes y servicios. 

El flujo está diseñado para emular la experiencia de un marketplace moderno 
(como PedidosYa o Airbnb), priorizando la intención de búsqueda exacta y 
ofreciendo alternativas (diversidad) cuando no hay coincidencias perfectas.

--------------------------------------------------------------------------------
1. COMPONENTES Y CLASES INVOLUCRADAS
--------------------------------------------------------------------------------

A. Schemas de Entrada/Salida (app/schemas/chat.py)
   - RecomendacionRequest: Recibe el mensaje del usuario y parámetros opcionales
     (fecha, distrito, presupuesto).
   - RecomendacionResponse: La respuesta estructurada devuelta al frontend.
     Contiene dos listas clave para la UX:
     * resultados_principales: Proveedores que hicieron "match" con la búsqueda.
     * otras_opciones: Proveedores que no hicieron match directo, pero se 
       muestran como sugerencias alternativas.

B. Modelos de Base de Datos (app/models/)
   - Proveedor, Paquete, DetallePaquete, ServicioProducto, OcupacionServicioProducto.

C. Servicios Core
   - ia_service.py: El orquestador o "Wrapper" que recibe el chat (pensado 
     para conectarse a un LLM en el futuro) y llama al motor determinístico.
   - recomendacion_service.py: El motor algorítmico principal.

--------------------------------------------------------------------------------
2. EL FLUJO PASO A PASO (recomendacion_service.py)
--------------------------------------------------------------------------------

El punto de entrada principal es la función `recomendar_evento()`.

--- FASE 1: Procesamiento de Lenguaje Natural Ligero (NLP) ---
1. _normalizar(texto):
   Pasa todo el texto a minúsculas y elimina tildes para evitar sensibilidad 
   a mayúsculas/minúsculas.
   
2. _detectar_intenciones(texto):
   Compara el texto contra el diccionario `KEYWORDS`.
   - Si dice "hora loca", detecta la intención "bailarina".
   - Si no detecta nada específico, asigna la intención "general" (sin sesgos).

3. _detectar_tematica_exacta(texto):
   Compara contra `TEMATICAS_CONOCIDAS` (Spiderman, Frozen, etc.).

4. _detectar_cantidades(texto):
   Usa Expresiones Regulares (Regex) para extraer cantidades solicitadas 
   (ej. "30 sillas" -> {"silla": 30}).

--- FASE 2: Selección y Puntuación (El Scoring) ---
El sistema itera sobre cada proveedor verificado. Por cada proveedor, ejecuta
`_elegir_paquete()` para determinar cuál de sus paquetes ofrecer y con qué puntaje:

Reglas de Puntuación (Score):
- +20 Puntos: Coincidencia de temática exacta (Ej: Pide "Spiderman" y el paquete es "Spiderman").
- -15 Puntos: Penalización cruzada (Ej: Pide "Frozen" y evalúa paquete "Spiderman". Evita sugerencias molestas).
- +5 Puntos: Coincidencia de palabras clave importantes (que no sean stopwords).
- +3 Puntos: El nombre de la intención ("dj", "animadora") está en el paquete.
- +2 Puntos: El paquete incluye servicios que coinciden con la intención.
- +1 Punto: Palabras genéricas coincidentes.

* Excepción Búsqueda Genérica: Si el usuario busca "quiero una fiesta" 
  (intención general), el motor fuerza la selección del paquete más barato 
  del proveedor y le da Score = 10 automáticamente para garantizar diversidad.

* Fallback de Seguridad: Si el proveedor no tiene ningún paquete que haga match 
  (Score final <= 0), el sistema forzosamente elige su paquete más barato y 
  le asigna Score = 0.

--- FASE 3: Clasificación UX (Principales vs Alternativas) ---
Con el `score_final` de cada proveedor, el sistema los divide en dos listas 
para el frontend:

- RESULTADOS PRINCIPALES (`score_final > 0`): 
  Opciones que realmente responden a lo que el cliente escribió.
- OTRAS OPCIONES / ALTERNATIVAS (`score_final == 0`): 
  Proveedores que se muestran para no tener "resultados vacíos", sirviendo 
  como sugerencias de exploración.

Ambas listas se ordenan por disponibilidad, luego por calificación y precio.

--- FASE 4: Construcción de la Propuesta ---
1. Adicionales Inteligentes:
   `_sugerir_adicionales()` revisa qué más necesita el cliente según su texto 
   y le sugiere items sueltos del proveedor que no estén ya en el paquete base.

2. Finanzas B2B:
   Se calcula el costo del paquete + adicionales. 
   El motor aplica matemáticamente la regla de negocio:
   - adelanto_20: 20% del total.
   - saldo_presencial: 80% restante a pagar en el evento.

3. Validación de Disponibilidad Real:
   `_validar_disponibilidad()` consulta la tabla `OcupacionServicioProducto`. 
   A diferencia de un query básico, usa una suma agregada (`func.sum`) para 
   sumar todos los cruces horarios y restarlos del `stock_maximo_simultaneo` 
   del servicio.

--- FASE 5: Respuesta Contextual de la IA ---
La función `_crear_respuesta()` genera un mensaje dinámico según los resultados:
- Si hay Principales: "Encontré X opciones ideales que coinciden..."
- Si solo hay Alternativas: "No encontré coincidencias exactas, pero te sugiero..."
- Si faltan datos: Solicita amablemente fecha, hora y dirección para permitir el prebloqueo.

--------------------------------------------------------------------------------
3. RENDERIZADO EN FRONTEND (React)
--------------------------------------------------------------------------------
En el archivo `App.tsx`:
- Los `resultados_principales` reciben un renderizado "Featured" (destacado) 
  con iconos dorados y bordes turquesa.
- Las `otras_opciones` se renderizan abajo en una grilla compacta de menor peso 
  jerárquico visual.
- Si ambas listas vienen vacías, se muestra un "Empty State" con un botón para
  intentar otra búsqueda.
